Agentic Interface

ProxyForge Agentic Interface#

ProxyForge must expose a simple, documented control surface for AI agents before the app is called Production Ready. This is intended for local operators who want Codex CLI, Claude CLI, and adjacent workspaces such as ~/vantix/ to drive authorized intelligence gathering, Chromium/browser capture, data collection, replay, Intruder runs, scanner planning, approved exploit workflows, callback polling, evidence packaging, report generation, and status export without guessing at UI internals.

Target Timing#

This work is scheduled after every MVP surface in the feature matrix is at least MVP Working, and before any Production Ready release claim.

Required Surfaces#

  • A stable headless command set for agents to launch ProxyForge tasks, inspect state, run approved workflows, and export artifacts.
  • Markdown instructions that explain the contract for Codex CLI, Claude CLI, and Vantix-side agents.
  • Machine-readable manifests for available tools, safety limits, report paths, and current project state.
  • Read-only status commands for inventory, history, findings, scans, callbacks, reports, and release validation artifacts.
  • Controlled action commands for scoped crawl, Chromium launch/cookie capture, proxy import, Sequencer token analysis/soak packages, Decoder transform chains, live target profiling, target map comparison, replay execution, bulk replay, replay matrix comparison, WebSocket list/replay/fuzz/transcript export, Intruder execution/soak packages, Repeater desync/race proofing and race soak packages, scanner plan creation, OAST issue promotion, Anvil custom scan-check validation/package export, extension compatibility fixtures, callback polling, callback replay, approved exploit runs, evidence attachment, and report export.
  • Explicit safety gates for exploit execution, active scans, callback replay, traffic mutation, Chromium profile access, credential/cookie handling, and any workflow that sends traffic to a live target.
  • Phase-aware outputs: operational/executor commands preserve full tokens, keys, cookies, headers, and raw traffic; submission/reporting commands redact before artifacts leave ProxyForge for report use.
  • Integration examples for ~/vantix/ showing how to invoke ProxyForge from an external agent runner.
  • External-cwd execution: agents can call an absolute proxyforge-agent.mjs path from ~/vantix/ while ProxyForge resolves its runtime modules from its own app root.

Agents must be able to execute the same core workflows an operator would expect from the GUI, with explicit scope, approval, full-fidelity operational evidence, and audit metadata on every active command.

Implemented Files#

  • docs/agents/CODEX.md: Codex CLI workflow contract, commands, output schemas, and safety boundaries.
  • docs/agents/CLAUDE.md: Claude CLI workflow contract, command examples, and report handoff expectations.
  • docs/agents/VANTIX.md: How Vantix agents should call ProxyForge, consume evidence, and feed results back into Vantix workspaces.
  • docs/agents/SCHEMAS.md: JSON shapes for project status, traffic summaries, scan plans, callback packages, and report exports.
  • docs/agents/MVP_OPTION_AUDIT.md: mandatory post-MVP gate that verifies every operator option has an agent command, schema guidance, or a tracked parity gap before parity/production review begins.
  • scripts/proxyforge-agent.mjs: stable local CLI wrapper around existing headless/runtime engines.

Post-MVP Option Audit#

After the last feature-matrix row reaches MVP Working, run the audit in docs/agents/MVP_OPTION_AUDIT.md before starting parity/production promotion. That audit must confirm every command in this document is implemented, every remaining GUI/runtime option is reachable by agents or tracked as a parity gap, and Intruder plus Repeater desync/race commands are documented for Codex, Claude, Vantix, and schema consumers.
